**Out of scope (explicitly deferred):**

<decisions>
## Implementation Decisions

### Postgres Tier + Cost (user-decided)

**Committed: Railway Hobby Postgres** — ~$5/mo shared compute, elastic storage, weekly automated snapshots, ~100 max connections, stays inside Railway private network (matches P0.5 "single-vendor compute" decision).

**Rationale (per user):** "The PII protection is provided by column-level encryption + Access + audit log, not by the Postgres tier. Hobby is fine while the app is pre-production."

**Upgrade trigger (documented in STATE.md Key Decisions):** Upgrade to Railway Pro when (a) Carrie's data becomes authoritative for daily operations, OR (b) the written GLBA Safeguards program requires daily DB backups — whichever comes first. Expected timing: ~Phase 5.

**Network:** DB traffic stays inside Railway's private network — no internet egress from app→DB. Preserves the P0.5 trust-boundary decision.

**Ruled out:**
- **Railway Pro** ($20/mo + daily snapshots + SOC2 on paper): breaks $15-20 budget; SOC2 not required for pre-authoritative internal tool
- **Neon free** ($0 + 24h PITR): PITR is attractive but expands trust boundary (Railway → internet → Neon); revisit if bulk-delete regret happens
- **Supabase free** ($0 + auto-pause): 1-week inactivity pause → 5-10s cold start on first request; violates 5-second-glance UX

### Seed Auto-Run Policy (user-decided)

**Committed: Option B — migrations auto, seeds manual.**

- `package.json` `prestart` script runs `npm run db:migrate` only
- Seeds run explicitly via `railway run npm run db:seed` when seed files change
- P0.5.1 playbook includes ONE-TIME initial seed step with post-seed verification

**Rationale (per user):** "Seeds modify data and data modifications in a GLBA-scope system need deliberate human action + audit trail. `onConflictDoUpdate` protects against duplicate inserts, not against bugs in the seed file itself."

**Initial seed verification (user-added):** After the one-time `npm run db:seed` runs against prod, read back `tracks` and `stages` tables, diff against the source `db/seed/tracks.ts` + `db/seed/stages.ts` files, log the result to `.planning/INITIAL_SEED_VERIFICATION.md` with date and outcome. Converts the initial seed from an invisible event into a documented, traceable one.

**Upgrade path:** If manual seed steps become frequent (e.g., P10 calibration adds many stage tunings), flip to auto by changing one line in `package.json` prestart script.

### Migration Hook Style (Claude's discretion — documented default)

**Committed: `package.json` `prestart` script.**

```json
{
  "scripts": {
    "prestart": "npm run db:migrate",
    "start": "next start"
  }
}
```

**Rationale:**
- Platform-agnostic — if Railway is ever replaced by Fly/Render/self-host, prestart keeps working. `railway.json postdeploy` would need rewrite on migration.
- Runs on EVERY container boot (initial deploy, crash recovery, scale-up) — auto-heals schema drift on restart
- Idempotent migrations (proven in P1-01) make repeat runs cheap (<1s for noop via `drizzle-kit migrate`)
- **Safe failure mode:** If migration fails during prestart, container fails to start → Railway health check fails → Railway keeps old deploy running → no swap to broken state. This is the defensive default.

**Ruled out:**
- `railway.json` `postdeploy` hook: runs once per build, but if it fails AFTER the app has started, you get a half-migrated DB with an app expecting the new schema. Harder to reason about.
- Post-deploy manual migration: defeats the "push and it works" ethos of DEPLOY-09.

### Staging Environment (Claude's discretion — documented default)

**Committed: Single environment (production only) — defer staging to post-incident.**

**Rationale:**
- 2x Railway cost pushes over $15-20/mo budget target from PROJECT.md
- Single-operator product: staging catches team-coordination bugs, not single-user bugs. Low ROI.
- Existing pre-prod surfaces: `docker-compose` local Postgres + 170 unit tests + `tsc --noEmit` + `npm run build` + `npm run db:migrate` against local DB. Migration safety is proven locally before push.
- Revisit trigger: first real prod incident where staging would have caught the issue.

**Ruled out:**
- Separate `staging` Railway environment with its own Postgres: doubles cost, duplicates CF Access config, creates a drift source.

### SC5 Direct-Origin 403 Verification (Claude's discretion — deferred item)

**Test command:**
```bash
curl -I https://npr-dashboard-prototype-production.up.railway.app
```

**Accept if:** Response is HTTP 403, 404, connection refused, or timeout. All indicate Railway's Public Networking is NOT exposing the service beyond the Cloudflare Tunnel, which is the P0.5 design intent.

**Reject if:** Response is HTTP 200 + actual app content. That would mean Railway is serving public traffic directly, bypassing CF Access entirely — a GLBA-relevant hole.

**Document actual output** in `00.5.1-VERIFICATION.md` so the test is reproducible.

### Rollback Playbook (Claude's discretion — deferred item)

**Written into `.planning/DEPLOY.md` runbook as part of this phase. Three layers:**

1. **App-only rollback (most common):** Railway dashboard → `npr-dashboard-prototype` service → Deployments → previous SUCCESS deployment → "Rollback." Traffic swaps back to the old container. Works when a bad code deploy passes tests locally but fails in prod (rare but possible).

2. **Data rollback:** Railway dashboard → Postgres service → Backups → restore latest snapshot. Weekly-snapshot-acceptable during P0.5.1 → P4 because deals are non-authoritative pre-P10 (Carrie is still in ClickUp). After P10 calibration, upgrade Postgres tier for daily snapshots before Carrie fully migrates.

3. **Forward-compensating migration:** When rollback isn't possible (e.g., migration already applied + new code already shipped). Write a new `000N_revert_XYZ.sql` migration that undoes the problematic schema change. **Acknowledged limitation:** `drizzle-kit generate` doesn't auto-generate down migrations — you write the compensating SQL by hand or copy-inverse the forward SQL from the `0005_*.sql` file.

**Where this goes:** New file `.planning/DEPLOY.md` — written as part of one of this phase's plans. Becomes the ongoing production runbook for all future phases.

## Specific Ideas

### Post-seed verification (user-specified)

After the one-time `npm run db:seed` runs against prod Postgres:

1. Query prod `tracks` table: expect 8 rows (TE, FL, DP, PO, EC, SL, BL, GI)
2. Query prod `stages` table: expect 25 rows (4 universal with `track_id=NULL` + 10 TE + 11 FL)
3. Diff prod row content against source `db/seed/tracks.ts` + `db/seed/stages.ts` — fields like `label`, `default_priority`, `code`, `sort_order` should match exactly
4. Log result to `.planning/INITIAL_SEED_VERIFICATION.md` with:
   - Date of seed run
   - Prod commit SHA at time of seed
   - Row counts (actual vs expected)
   - Field-level diff (if any)
   - Outcome: "verified" | "mismatch — investigated and resolved" | "mismatch — escalated"

This converts the invisible "seed ran" event into a documented artifact that proves prod is correctly initialized.

### Verification of CFA-11 (direct-origin 403)

Test from ANY machine (not inside Railway): `curl -I https://npr-dashboard-prototype-production.up.railway.app`. Record exact response code + any headers in `00.5.1-VERIFICATION.md`. If 200 + app content, that's a P0.5 regression — escalate.

### Verification of CFA-12 (avatar shows real user)

After CF Access login at `portal.utstitle.com`:
1. Open browser dev tools → Application → Cookies → note CF_Authorization presence
2. Inspect the `AppHeader` avatar component (`components/app-header.tsx`)
3. Confirm it renders the authenticated user's name/email from `lib/current-user.ts` — NOT the fallback `"CD"` initials
4. Record screenshot + user.email payload in `00.5.1-VERIFICATION.md`

<deferred>
## Deferred Ideas

### Staging environment
Adding a second Railway `staging` environment. Deferred because single-operator + budget + existing pre-prod surfaces (docker-compose, 170 tests, tsc, build) cover most risks. Revisit after first prod incident where staging would have caught it.

### Auto-run seeds on deploy
Flipping to `prestart: npm run db:migrate && npm run db:seed`. Deferred because seeds modify data and need deliberate human action in a GLBA-scope system. One-line change when warranted — e.g., if P10 calibration starts pushing seed tunings frequently.

### Railway Pro tier upgrade
Moving to $20/mo Pro for daily snapshots + SOC2 attestation. Deferred; trigger documented in STATE.md: (a) Carrie's data becomes authoritative for daily ops, OR (b) written GLBA Safeguards requires daily DB backups. Expected timing ~P5.

### Neon / Supabase migration
External Postgres providers with better features (PITR on Neon free). Deferred because it expands trust boundary and breaks P0.5 single-vendor decision. Revisit only if a specific feature becomes essential.

### Cloudflare Access policy fix for `mike@`/`info@`
Tracked separately in `.planning/debug/2026-04-17-portal-access-denied.md`. Blocks end-user UAT of P1+P2 but NOT this phase's success criteria (which test the deploy topology, not Carrie's login experience). Address in parallel.

### `railway.json` declarative config
Using a `railway.json` file in the repo to declare the prestart hook + other Railway config. Deferred in favor of `package.json prestart` for platform-agnosticism. Revisit if Railway-specific features (like the `deploymentConfig` block) become needed for future phases.

### Reverse migrations (down SQL)
`drizzle-kit` doesn't auto-generate down migrations. Hand-writing compensating migrations for each forward migration would be a substantial process investment. Deferred — the runbook documents the forward-compensating strategy, which suffices for the scale of changes v1 ships.

</deferred>
