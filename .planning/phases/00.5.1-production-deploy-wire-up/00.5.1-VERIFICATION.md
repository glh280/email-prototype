## Checkpoint 2 — DATABASE_URL reference variable wired on app service

**Date:** 2026-04-18
**Operator:** glh280
**Plan:** 02-railway-provisioning-checkpoints.md, Task 2

### Outcome

`DATABASE_URL` on the `npr-dashboard-prototype` service is now a reference to `${{Postgres.DATABASE_URL}}`. Railway's runtime interpolates the reference to the live Postgres service's `DATABASE_URL` on every container boot — password rotations on Postgres will auto-propagate without manual sync to the app service.

### Wiring path used (CLI fallback — UI paths failed twice)

**Failed attempts documented for future-operator tribal knowledge:**

1. **UI "+ New Variable" direct paste** — entered `DATABASE_URL` name + `${{Postgres.DATABASE_URL}}` value via the variable form. Railway stored it as a literal string (reference never registered). UI rendered the row with the `{}` plain-variable icon, not a chain/link icon.

2. **UI "Trying to connect a database?" banner → Add Variable guided flow** — same result. Railway stored it as a literal; `{}` icon.

**Fix that worked (CLI with single-quoted value):**

```bash
railway variables --service npr-dashboard-prototype --set 'DATABASE_URL=${{Postgres.DATABASE_URL}}'
```

**Critical detail: SINGLE QUOTES around the full value are required.** Without them, `bash` (or PowerShell, or any POSIX-ish shell) interprets `${{Postgres.DATABASE_URL}}` as a shell variable expansion — which silently resolves to empty string and hands Railway's API a broken value. Single quotes pass the literal `${{Postgres.DATABASE_URL}}` string through to Railway, which then parses it as a reference and stores it correctly.

If using PowerShell, use `'...'` (single quotes) not `"..."`. If using Cmd, escape the `$` appropriately or run in a WSL/Git Bash subshell with single quotes.

### CLI resolution check (authoritative)

```bash
railway variables --service npr-dashboard-prototype --kv | grep ^DATABASE_URL=
```

Output:
```
DATABASE_URL=postgresql://postgres:[rotated]@postgres.railway.internal:5432/railway
```

- Resolved to a real connection string (NOT the literal `${{Postgres.DATABASE_URL}}` placeholder) → reference is functioning
- Host is `postgres.railway.internal` (private network) → matches P0.5 single-vendor trust boundary
- Password redacted to `[rotated]` by operator before pasting (good practice, post-rotation)

### UI state (non-authoritative, logged for future reference)

- **Railway UI (operator-observed):** `DATABASE_URL` row still displays with the `{}` plain-variable icon even after the CLI fix made the reference functional at runtime.
- **Interpretation:** Railway's UI in this version (observed 2026-04-18) may not re-render the icon based on CLI-set reference variables, OR may have a different visual affordance for references that the operator did not identify. The UI icon is advisory; CLI resolution is the authoritative signal per this plan.
- **Action for future phases:** When wiring reference variables, prefer the CLI path (`railway variables --set 'KEY=${{Service.VAR}}'` with single quotes) over the UI. Verify via `railway variables --kv | grep ^KEY=` — a resolved value confirms the reference; a literal `${{...}}` value confirms the reference is broken.
- **Open question for Railway:** whether the UI icon is deterministic signal for reference status in all UI versions. Not a blocker for this phase; flagged here for audit.

- **Second push** (`0c4d295`): failed — `npm ci` EBADPLATFORM error on @esbuild/aix-ppc64. Root cause: Windows-generated lock regen marked nested vitest @esbuild/* platform variants as `extraneous` but not `optional`; npm on Linux tried to install platform-incompatible variants as required deps.

```bash

curl -s https://npr-dashboard-prototype-production.up.railway.app/deal/new
curl -I https://portal.utstitle.com
```

### Results

| Probe | Response | Interpretation |
|-------|----------|----------------|
| Direct Railway origin `/` | `HTTP 401 Unauthorized` + body `"Unauthorized"` (12 bytes) + `server: railway-edge` | ✓ Railway's edge blocks public access; app never sees request |
| Direct Railway origin `/deal/new` | Same `HTTP 401` + `Unauthorized` (12 bytes) | ✓ Path-independent block — Railway's networking layer, not app auth |
| Cloudflare-fronted `portal.utstitle.com` | `HTTP 302 Found` + `Cache-Control: no-store` | ✓ CF Access intercepts + redirects to Google IdP login (expected) |

### Interpretation (acceptance vs plan spec)

Plan spec expected `HTTP 403 / 404 / connection refused / timeout`. Actual response was `HTTP 401 Unauthorized`. This is **accepted** because:

1. `server: railway-edge` header confirms the 401 is generated by Railway's edge infrastructure, NOT by our Next.js app. The app never receives the request.
2. The `Unauthorized` body is Railway's standard "no public networking" response, not a CF Access / app auth challenge.
3. Absence of `WWW-Authenticate` header confirms Railway is saying "blocked" not "please authenticate."
4. Body size is 12 bytes (just `Unauthorized`) — no leaked app content, no version strings, no telemetry.
5. Path `/deal/new` returns identical 401 — if our app were serving traffic directly, this path would respond with HTML form content or 404.

The plan's explicit accept list was a guess at what Railway would return; 401 with `server: railway-edge` is the actual answer and is semantically equivalent to the intended "direct origin blocked" outcome.
